## 4. 検索精度評価

### 4.1 hybrid search (FTS + vector + RRF)

| クエリ | top-1 書籍 | top-1 章 | 妥当性 |
|---|---|---|---|
| ドメインモデリング 値オブジェクト | データモデリング本 | 第4部 ファウンデーション | OK |
| 本を書くときに文章をわかりやすくするコツ | テクニカルライティング | 第3章 読み手に伝わる文章 | OK |
| ソフトウェア設計におけるデータの扱い方 | データモデリング本 | 第2部 実践 | OK |
| 技術文書の書き方のベストプラクティス | テクニカルライティング | 第2章 わかりやすく簡潔な文章 | OK |
| 文章 (短いキーワード) | テクニカルライティング | Column「文、文章、文書」の違い | OK |

### 4.2 FTS (trigram) の特性

- 3文字以上のキーワード単体では正常にヒット
- 長い自然文クエリでは全 trigram の AND 条件となり 0 件になりやすい
- hybrid モードでは vector 検索が補完するため実用上問題なし
- ただし hybrid の横断性が vector に依存するため、1冊に集中する傾向あり

### 4.3 vector search の特性

- 意味的に近い書籍・章を的確にランク付け
- 類似度スコアは 0.91〜0.99 の範囲（cosine distance）
- 同一書籍内のチャンクが上位を独占しがちで、横断性に課題

## 7. 発見された課題と本格実装への提言

### 7.1 FTS の日本語対応 (優先度: 高)

**課題**: trigram トークナイザは CJK の最小限サポート。長い日本語クエリで 0 件になる。2文字のキーワードも検索不可。

**提言**: MeCab 等の形態素解析トークナイザを SQLite FTS5 に組み込む。または、Go 側でクエリを形態素分解してから FTS に渡す前処理を追加。

### 7.2 横断検索の多様性 (優先度: 中)

**課題**: vector 検索が最も近い 1 冊のチャンクで上位を独占。複数冊比較のユースケースが弱い。

**提言**:
- 検索結果に「書籍ごとの上限」を設けて多様性を確保（例: 1冊あたり最大 2 チャンク）
- Maximal Marginal Relevance (MMR) によるリランキング導入

### 7.3 EPUB テキストクリーニング (優先度: 中)

**課題**: 一部 EPUB でレイアウト由来の不要改行・装飾文字がチャンクに残存。章タイトルが1文字ずつ改行される等。

**提言**: Python worker のテキスト後処理を強化。連続する1文字改行の除去、装飾記号の除去ルールを追加。

### 7.4 重複 ingest 防止 (優先度: 低)

**課題**: 同じ書籍を異なるパスから ingest すると重複登録される（source_path が UNIQUE キー）。

**提言**: file_hash による重複検出を追加。同一ハッシュの書籍が既にある場合はスキップ。

### 7.5 画像 PDF 対応 (優先度: 低・PoC 対象外)

**課題**: テキストレイヤーのない PDF（スキャン画像）は抽出不可。

**提言**: 本格実装では OCR (Tesseract or Apple Vision Framework) をオプションで対応。

### 7.6 チャンクサイズの最適化 (優先度: 低)

**課題**: 現在 500 文字固定。書籍によって最適なサイズが異なる可能性。

**提言**: ingest 時にチャンクサイズを書籍ごとに調整可能にする。reindex で試行錯誤できる現在の仕組みは有効。
