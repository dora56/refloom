# ADR-0012: 反復的クエリ書き換え

- Status: Proposed
- Date: 2026-03-28
- Deciders: dora56

## Context

現在の `ask` コマンドは 1 回の検索で取得したチャンクをそのまま LLM に渡す。
複雑な質問や曖昧なクエリでは、1 回の検索で十分なコンテキストが得られないことがある。

例: 「ドメインモデリングと関数型プログラミングの共通点は？」
→ validate スコアで keyword/hybrid ともに MISS (q08)。
   1 回の検索では片方の書籍のチャンクしかヒットしない。

エージェンティック RAG の研究 [3,4,5] では、検索を「状態依存の繰り返しアクション」として扱い、
前の検索結果に基づいてクエリを動的に書き換える反復的検索が有効であることが示されている。
Self-RAG [1,5,6] はモデルが「いつ検索が必要か」を判断し、取得情報の関連性を評価する。

## Decision

**`ask` コマンドに最大 2 回のクエリ書き換えループを追加する。**

### 実装方針

1. 初回検索を実行し、結果を LLM に渡す
2. LLM に「この結果で質問に十分答えられるか」を判断させる (Self-RAG 判定)
   - 十分 → そのまま回答生成
   - 不十分 → LLM が改善クエリを生成
3. 改善クエリで 2 回目の検索を実行、結果をマージ
4. 最大 2 ラウンド (初回 + 2 回書き換え = 最大 3 回検索)
5. `--no-refine` フラグで無効化、`--max-refine N` で回数制御

### プロンプト設計

```
あなたは検索結果の十分性を評価するアシスタントです。
ユーザーの質問と検索結果を分析し、以下の JSON を返してください:
{
  "sufficient": true/false,
  "reason": "不足している情報の説明",
  "refined_query": "改善されたクエリ (sufficient=false の場合のみ)"
}
```

### 検討した選択肢

| 選択肢 | 判定 | 理由 |
|--------|------|------|
| LLM 判定 + クエリ書き換え (最大 2 回) | **採用** | 精度向上と制御可能性のバランス |
| 固定 2 回検索 (常にクエリ変換) | 不採用 | 不要な場合もレイテンシ増加 |
| 無制限ループ | 不採用 | コスト・レイテンシの予測不可能性 |
| HyDE (仮説生成→検索) | 不採用 | 追加 LLM 呼び出しのコスト。書き換えの方が汎用的 |

## Consequences

### Positive

- cross-book 質問 (q08, q11) のスコア改善が期待できる
- LLM が自律的に検索の必要性を判断 → 不要なケースではオーバーヘッドなし
- `--no-refine` で従来動作を維持可能

### Negative

- 最大 3 回の検索 + 2 回の判定 LLM 呼び出し → レイテンシ最大 3-5 倍
- LLM 判定の品質がモデルに依存 (claude-cli vs ollama)
- claude-cli 使用時の API コスト増加

### Neutral

- validate スコア (keyword/hybrid) への影響は `ask` のみ。search は変更なし
- 既存の citation budget 制御は維持。マージされた結果に対して budget を適用
