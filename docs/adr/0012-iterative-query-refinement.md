# ADR-0012: 反復的クエリ書き換え

- Status: Accepted (HyDE variant)
- Date: 2026-03-28
- Accepted: 2026-03-30
- Deciders: dora56

## Context

現在の `ask` コマンドは 1 回の検索で取得したチャンクをそのまま LLM に渡す。
複雑な質問や曖昧なクエリでは、1 回の検索で十分なコンテキストが得られないことがある。

例: 「ドメインモデリングと関数型プログラミングの共通点は？」
→ validate スコアで keyword/hybrid ともに MISS (q08)。

既存の `DetectIntent` + `DiversifyByBook` による比較クエリ対応は動作しているが、
q08/q11 では対象書籍のチャンクが初回検索で上位にランクしないことが根本原因。
Diversification は候補プール内の並べ替えであり、候補自体が不足する問題は解決しない。

エージェンティック RAG の研究では、前の検索結果に基づいてクエリを動的に書き換える
反復的検索が有効であることが示されている。

## Decision

**`ask` コマンドに最大 1 回のクエリ書き換えを追加する（合計最大 2 回検索）。**

### 実装方針

1. 初回検索を実行し、結果を LLM に渡す
2. LLM に「この結果で質問に十分答えられるか」を判断させる
   - 十分 → そのまま回答生成
   - 不十分 → LLM が改善クエリを生成
3. 改善クエリで 2 回目の検索を実行、結果をマージ（chunk_id で重複排除）
4. `--max-refine N` フラグで制御（デフォルト 1、0 で無効化）

### 充足性判定の実装

**課題**: 現在の `llm.Provider` は `Generate(ctx, system, user string) (string, error)` のみで
JSON 構造化出力を保証する仕組みがない。

**対応策**:
- テキスト応答の先頭行から判定を抽出するシンプルなプロンプト設計
- 例: 「SUFFICIENT」or「REFINE: <改善クエリ>」の 1 行目を判定に使用
- JSON パースではなく文字列プレフィックスマッチで実装（モデル非依存）

### レイテンシとタイムアウト

現状の `ask` タイムアウトは 60 秒 (config)。claude-cli の generation は ~15 秒。

| パターン | 検索 | LLM 呼び出し | 推定合計 |
|---------|------|-------------|---------|
| 1 回で十分 | 1 回 (~0.2s) | 1 判定 + 1 生成 (~30s) | ~30s |
| 書き換え 1 回 | 2 回 (~0.4s) | 1 判定 + 1 生成 (~30s) | ~30s |
| **最悪ケース** | 2 回 (~0.4s) | 1 判定 + 1 生成 (~30s) | **~30s** |

判定と生成を 1 回の LLM 呼び出しに統合（「不十分な場合は改善クエリを提案しつつ、
可能な範囲で回答も生成」）することで、追加 LLM 呼び出しを最大 1 回に抑える。

### 検討した選択肢

| 選択肢 | 判定 | 理由 |
|--------|------|------|
| 判定統合型 1 回書き換え (最大 2 検索) | 不採用 | HyDE が q08 を改善したため不要 |
| HyDE (仮説回答→検索) | **採用** | 語彙ミスマッチに強い。1 LLM 呼び出しで完結。PoC で q08 改善を確認 |
| 固定 2 回検索 (常にクエリ変換) | 不採用 | 不要な場合もレイテンシ増加 |
| 無制限ループ | 不採用 | コスト・レイテンシの予測不可能性 |

**HyDE との比較**: HyDE は仮説回答を生成して embedding 検索に使う手法で、語彙ミスマッチ (q08/q11) に
特に有効。LLM 1 回呼び出しで済むため本提案よりコスト効率が高い可能性がある。
PoC では両方を実装・比較して効果を検証する。

## Consequences

### Positive

- cross-book 質問 (q08, q11) のスコア改善が期待できる
- LLM 判定を生成に統合 → 追加呼び出しは最大 1 回
- `--max-refine 0` で従来動作を維持可能

### Negative

- 最悪ケースでレイテンシ ~2 倍（検索 2 回 + 判定統合型 LLM 1 回 + 最終生成 1 回 ≈ 30s）
- LLM 判定の品質がモデルに依存（小型モデルでは判定精度が低い可能性）
- claude-cli 使用時の API コスト ~2 倍（最悪ケース: LLM 2 回呼び出し）

### Neutral

- validate スコア (keyword/hybrid) への影響は `ask` のみ。search は変更なし
- PoC で HyDE との A/B 比較を実施し、効果の高い方を本採用する

## PoC 結果 (2026-03-30)

**HyDE を採用。反復書き換えは不要と判断。**

### HyDE 実装

- `--hyde` フラグで LLM に仮説回答を生成 → embedding → 3-way RRF マージ (FTS + original vector + hypothesis vector)
- `SearchHybridWithHyDE` メソッドと variadic `reciprocalRankFusion` で 3 ソース統合

### 検証結果

| Query | Mode | Books retrieved | 判定 |
|-------|------|----------------|------|
| q08 | baseline | 関数型DM, テクライ | MISS (データモデリング本なし) |
| q08 | **hyde** | **関数型DM, テクライ, データモデリング** | **HIT** |
| q11 | baseline | 関数型DM, テクライ, SaaS | MISS |
| q11 | **hyde** | 関数型DM, テクライ, SaaS | MISS (チャンク品質問題) |

- q08: HyDE で「データモデリングでドメインを駆動する」のチャンクを取得成功。語彙ミスマッチ解決
- q11: 改善なし。索引・目次チャンクしか存在しないためチャンク品質の問題
- レイテンシ: +13s (LLM 仮説生成 1 回分)。total ~28s で 60s timeout 内
- validate スコア: keyword 9/12, hybrid 9/12 (リグレッションなし)
