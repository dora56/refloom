# 100冊スケール見積もり

## 現状 (5冊)

| 指標 | 値 |
|---|---|
| 有効書籍数 | 5冊 (PDF 3, EPUB 2) |
| チャンク数 | 3,141 |
| テキスト量 | 1.85 MB |
| DB サイズ | 31.7 MB |
| Search レイテンシ (hybrid) | 0.15〜2.2秒 |
| Ingest 所要時間/冊 | 約 2.5分 (embedding 含む) |

## 100冊線形外挿

| 指標 | 推定値 | 根拠 |
|---|---|---|
| チャンク数 | ~63,000 | 3,141 × 20 |
| テキスト量 | ~37 MB | 1.85 × 20 |
| DB サイズ | ~634 MB | 31.7 × 20 (embedding が支配的) |
| Ingest 総時間 | ~4時間 | 2.5分 × 100 |

## ボトルネック分析

### ベクトル検索 (sqlite-vec KNN)

- sqlite-vec `vec0` は brute-force KNN (インデックスなし、O(n) 全スキャン)
- 768次元 × 63,000ベクトル ≈ 193 MB のスキャン
- 推定: 1〜5秒/クエリ（現在の 0.1〜0.5秒から増加）
- **100冊ではまだ実用範囲内だが、500冊超で顕著に悪化する見込み**

### FTS 検索

- FTS5 B-tree インデックスは O(log n) でスケール
- 63,000 チャンクでも sub-second を維持する見込み
- **ボトルネックにはならない**

### Embedding 生成

- embeddinggemma (768次元) のスループット: ~20チャンク/秒 (Apple Silicon)
- 63,000チャンク ÷ 20 = ~52分 (初回全 ingest)
- **初回は時間がかかるが、差分 ingest なら問題ない**

### DB サイズ

- 634 MB は Apple Silicon Mac のストレージとしては許容範囲
- embedding が DB サイズの約 85% を占める

## 推奨アクション

### 100冊時点では不要

- ANN インデックス（IVF, HNSW 等）への移行
- embedding の外部ストレージ化
- 分散アーキテクチャ

### 100冊超で検討

| 対策 | タイミング | 効果 |
|---|---|---|
| sqlite-vec IVF インデックス | 500冊以上 | KNN を O(√n) に改善 |
| Embedding バッチ API | 100冊以上 | ingest 時間短縮 |
| 差分 embedding (未変更スキップ) | 即座に有効 | reindex 時間短縮 |
| DB 分割 (書籍単位) | 1000冊以上 | 並列検索・管理容易性 |

## 結論

現在のアーキテクチャは 100冊スケールで十分に動作する見込み。
主な制約は sqlite-vec の brute-force KNN だが、100冊 (63K ベクトル) では
数秒以内に収まると予想される。500冊超で ANN インデックスの導入を検討。
